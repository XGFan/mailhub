# @mailhub/portal-server — MailHub backend API

A Node.js/TypeScript server (Hono + Drizzle + pg) that:

1. **Polls R2** every 30 seconds to discover new mail buffered by the Worker
2. **Parses email** in an isolated worker thread (DoS protection, timeout)
3. **Stores to Postgres** with idempotency (exact-once via `r2_key` unique constraint)
4. **Serves REST API** for search, detail, attachments, settings
5. **Auto-purges** mail older than 7 days (configurable)

## Quick start (local dev)

### Prerequisites

- Node.js ≥ 20
- Docker + OrbStack (for Postgres + MinIO)
- pnpm

### 1. Start the local stack (Postgres 16 + MinIO)

```bash
docker compose -f docker-compose.yml up -d

# Verify services are healthy:
# - Postgres: localhost:5432
# - MinIO console: http://localhost:9001 (mailhub / mailhub-secret)

# Create the MinIO bucket:
# Visit http://localhost:9001, log in, and create bucket "mailhub-raw"
```

### 2. Set up the portal server

```bash
# Copy and edit .env (from the repo-root .env.example)
cp ../../.env.example .env
# Edit:
#   DATABASE_URL=postgres://mailhub:mailhub@localhost:5432/mailhub
#   R2_ENDPOINT=http://localhost:9000
#   R2_ACCESS_KEY_ID=mailhub
#   R2_SECRET_ACCESS_KEY=mailhub-secret
#   R2_BUCKET=mailhub-raw

# Install dependencies
pnpm install

# Run migrations (creates schema, tables, indexes)
pnpm db:migrate

# Start the dev server (listens on port 8787)
pnpm dev
```

The API is now available at `http://localhost:8787`.

### 3. Verify health

```bash
curl http://localhost:8787/healthz      # should return {"ok":true}
curl http://localhost:8787/readyz       # should return checks.db=true, checks.r2=true
curl http://localhost:8787/api/mails    # should return empty search results
```

```bash
pnpm test                           # run all tests
TEST_DATABASE_URL=postgres://... pnpm test  # override test DB
```

Key tests:
- **`ingest-idempotency.test.ts`:** Verifies re-processing the same mail twice doesn't create duplicates
- **`search.test.ts`:** Validates search params and SQL injection prevention
- **`sanitize.test.ts`:** Checks HTML sanitization output
- **`path-safety.test.ts`:** Ensures attachment paths can't traverse ATTACHMENT_DIR

### Database migrations

```bash
pnpm db:migrate              # apply pending migrations (auto-runs on startup)
pnpm db:generate <migration-name>  # create a new migration
```

Migrations are in `drizzle/` directory (auto-generated by Drizzle).

## Configuration

All settings are environment variables (see `.env.example`):

| Variable | Required | Default | Notes |
|----------|----------|---------|-------|
| `DATABASE_URL` | Yes | — | Postgres connection (least-priv role) |
| `R2_ENDPOINT` | Yes | — | R2 S3-compatible endpoint |
| `R2_ACCESS_KEY_ID` | Yes | — | R2 API token access key |
| `R2_SECRET_ACCESS_KEY` | Yes | — | R2 API token secret key |
| `R2_BUCKET` | No | `mailhub-raw` | R2 bucket name |
| `PORT` | No | `8787` | HTTP listen port |
| `POLL_INTERVAL_MS` | No | `30000` | Ingestor poll interval (ms) |
| `ATTACHMENT_DIR` | No | `./data/attachments` | Where attachment files are written |
| `MAX_MAIL_BYTES` | No | `27262976` | Reject messages larger than this |
| `RETENTION_DAYS` | No | `7` | Auto-purge mail older than N days |

## Architecture

### Ingestor (singleton loop)

Runs every `POLL_INTERVAL_MS` (default 30s):

1. **LIST:** Paginated `ListObjectsV2({Prefix: "inbox/", MaxKeys: 100})`
2. **GET:** Fetch each object (with size cap)
3. **Parse:** In isolated worker thread (timeout + memory limit)
4. **Store:** `INSERT ... ON CONFLICT (r2_key) DO NOTHING` (idempotency)
5. **Attachments:** Write parsed attachments to PVC
6. **Delete:** Remove from `inbox/` after successful commit
7. **Backoff:** Exponential backoff on consecutive errors (prevents hammering)

Failed parses move to `dead/` (visible in R2 console).

### API server (Hono)

Handles:
- **Search:** Parameterized, validated, rate-limited
- **Download:** Forced-download headers, path safety checks
- **Security:** CSP/X-Frame-Options/nosniff headers on all responses
- **Probes:** Liveness (`/healthz`) and readiness (`/readyz`)

### Auto-purge scheduler

Runs daily (in-process interval):
- Deletes rows where `received_at < now() - RETENTION_DAYS`
- Deletes corresponding attachment files from PVC

## Database schema

See `src/db/schema.ts` for the ORM definition. Key tables:

**`mails`**
- `id` (UUID, PK)
- `r2_key` (TEXT, UNIQUE) — idempotency anchor
- `message_id` (TEXT) — Message-ID header (non-unique, indexed for threading)
- `to_addr`, `from_addr`, `subject` (TEXT) — envelope/header info
- `date` (TIMESTAMPTZ, nullable) — parsed header date
- `received_at` (TIMESTAMPTZ, NOT NULL) — derived from r2_key epoch
- `html_sanitized`, `text_body`, `snippet` (TEXT)
- `has_attachments`, `is_spam` (BOOLEAN)
- `auth_results` (TEXT) — raw Authentication-Results header
- `raw_path` (TEXT) — path to archived .eml on PVC

**`attachments`**
- `id` (UUID, PK)
- `mail_id` (UUID, FK) — cascade delete
- `filename`, `mime_type` (TEXT)
- `storage_path` (TEXT) — server-generated UUID path (never from filename)
- `content_id` (TEXT) — for inline `cid:` parts
- `is_inline` (BOOLEAN)

**`settings`**
- `id` (INTEGER, PK = 1)
- `show_remote_images` (BOOLEAN)

**Indexes:**
- GIN `pg_trmg` on `to_addr`, `from_addr`, `subject` (substring search)
- B-tree on `date DESC`, `received_at DESC` (ordering)

## Production notes

### Monitoring

- **Ingest lag:** `oldest(inbox/) age` should stay < 60s. Alert if > 2 × POLL_INTERVAL.
- **Parse failures:** Count objects in `dead/` bucket. Monitor for spikes.
- **PVC usage:** Attachments grow ~500 MB/day per 1,000 mails. Monitor `df -h /data/attachments`.
- **Postgres:** Monitor locks and long-running queries (especially search on large result sets).

### Scaling (future)

Phase 1 is a singleton deployment (ingestor + API in one Pod). To scale the API:

1. **Split ingestor:** Move to a separate StatefulSet (single replica) with leader election
2. **Scale API:** Deploy multiple stateless API replicas, share Postgres
3. **Load balance:** Use k8s `Deployment` with `replicas: N` and DNS round-robin

Current single-pod design is fine for up to ~10,000 mails/day.

### Backup & recovery

- **Postgres:** Backup schema regularly (dump SQL, or use `pg_basebackup`)
- **Attachments PVC:** Use k8s volume snapshots or external backup
- **Raw .eml archive:** Optional — all emails are transient in R2 (deleted after parse) unless you configure R2 lifecycle retention
